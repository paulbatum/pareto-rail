# Benchmark overhaul: goal, principles, and execution plan

Status: complete. This document is the record of the v2-era simplification of the benchmark system — why it happened, the principles that governed every decision, and what landed. Everything past phase 5 is launch preparation, decided by the owner.

## 1. Where this started

The trigger was a simple question: what is the practical difference between a rehearsal run and a real (eligible) benchmark run? Examining the two mass-driver rehearsals (`rehearsal-mdd-fable-4m9v`, `rehearsal-mdd-sol-7k2p`) showed that the execution pipeline was identical — same checkout, rendered assignment, stage launch, gates, sealed commit, payload, cost capture. What rehearsals skipped was the surrounding apparatus that eligible runs required: a frozen release record and git tag, hash-pinned run definitions, a cryptographically shuffled private schedule, frozen controller-code hashes, and pre-declared eligibility.

That prompted the real question: is this system over-engineered? The owner's original goal was narrow — **do not accidentally vary the instructions between benchmark runs** (the assignment prompt, AGENTS.md, the level-building guidance an entrant reads). The apparatus had grown far beyond that goal, to the point of working against it: during the Sol mass-driver rehearsal, a bug in the scope checker nearly turned an expensive completed run into a DNF, and fixing the bug mid-run required a baseline swap — because the scope *tooling* was frozen together with the entrant *instructions*. Meanwhile the machinery that would have prevented harmless tooling fixes (`PARETO_RAIL_ACCEPT_CONTROLLER_DRIFT`, controller-admin hash freezing, runner/executor pinning) protected nothing the owner cared about.

The diagnosis: the system was designed on two false premises. First, that the audience is adversarial — that every claim ("inputs were identical", "we didn't cherry-pick", "ranking was blind") needs machine-checkable pre-proof against a hostile auditor. The actual audience is the owner and site visitors having fun. Second, that the operator has no judgment — the 250-line runbook, dispatcher isolation rules, and frozen failure taxonomy were written so an agent could execute the protocol without thinking. The actual operators are the owner and an agent working with the owner; judgment is available and cheap.

Two related decisions from the same conversation are recorded here because they shape v2:

- **Theme reuse is the owner's call, not a protocol rule.** `mass-driver-detailed` exists deliberately as the detailed arm of a pair with `mass-driver`, to measure whether richer specs close the quality gap; the rehearsals on it do not disqualify it from v2. The entrant-side contamination argument (prior solutions reachable in the repo) was real but is now solved by mechanism — see isolated checkouts below. The judge-side argument (the owner has seen rehearsal outputs) was rejected as negligible for blind pairwise voting across a full pool.
- **Entrants do not need git history.** Sharing the repository's full history with entrant sessions bought one saved `git fetch` per run and cost a permanent policy burden ("non-adversarial access", theme-burn worries, disclaimer prose). Isolation by construction replaced it.

## 2. Goal

A benchmark system a person can hold in their head. Version 2 runs execute under a protocol whose honesty comes from **captured artifacts** — the rendered prompt, the frozen baseline commit, the sealed entrant commit, the transcript, the measured cost — rather than from ceremony. Everything whose only purpose is defending claims nobody is attacking gets deleted, and v2 runs start on the simplified system.

## 3. Principles

1. **Ceremony proportional to cost.** Generation runs are expensive and unrepeatable — their inputs (prompt, baseline, recipe) deserve strictness. Gates, bookkeeping, and publishing are deterministic and replayable — they deserve at most a recorded commit hash.
2. **Capture, don't pre-verify.** A run is honest because its artifacts are durably kept. Any fairness question is answerable after the fact by diffing artifacts. Pre-launch verification shrinks to what prevents wasting an expensive run — chiefly: sibling runs of one theme must receive byte-identical base prompts.
3. **Mechanism over policy; judgment over taxonomy.** Where a worry can be eliminated by construction (isolated entrant checkouts), do that and delete the policy prose. Where it cannot, the operator decides and writes a note in the run record — no pre-declared classification systems.
4. **The only load-bearing blindness is the voter's — and the owner is a voter.** Site visitors vote on anonymous level pairs, enforced mechanically by the site. The owner votes through the same site and stays blind by discipline until they have voted: a helper assigns random slot ids when expanding the plan (so authoring it doesn't reveal the mapping), the owner doesn't open the plan file, and operator surfaces (results, status, promotion) print no configuration identity by default, with an explicit unblind flag for after voting. Level ids stay opaque (`<theme-id>-<slot>`), and the level→configuration mapping lives only in the private plan. What stays deleted: schedule cryptography, dispatcher isolation, and every mechanism whose only job was *enforcing* what discipline and defaults already achieve.
5. **The controller is a program, not a supervised agent.** `scripts/benchmark/run.mjs` is the pipeline. Documentation tells a human which command to run and what to do when it fails — one page, not a constitution.
6. **Failure policy is two sentences.** Infrastructure failure: fix, rerun, keep and report the cost of every attempt. Model failure (a gate fails on the sealed output): DNF, shown as such. The operator classifies and records the decision as a free-text note.
7. **v1 is finished, not a living constraint.** Its published levels and records are history; its machinery is retrievable at the `benchmark-v1` git tag. Live code supports v2 only. Historical records stay readable where cheap (the results listing); dual code paths are deleted.

Hard constraints that bound all of this: the public site keeps working (`docs/compat.md` contracts, the published rank catalog's v1 slice, domain/catalog tests); cost measurement (ccusage, reconciliation, budget machinery) is real measurement and stays fully intact; checkpoints, `--resume`, and recovery snapshots stay (cheap insurance protecting expensive runs); nothing under `benchmark/private/` is rewritten; theme, recipe, and prompt files are verbatim payload and are never edited by tooling work.

## 4. Work completed (with commits)

The first two steps predate the "delete it all" decision — they refined the freeze rather than removing it — but their surviving pieces are noted.

- **`6d50cd9` — Split the protocol into frozen entrant inputs and recorded controller tooling** (docs). Established the tier principle: freeze what an entrant sees or is judged by; record controller tooling as provenance. Introduced the replay argument — gates are deterministic against sealed commits, so tooling fixes are applied by re-running gates, never by re-running generation. The freeze concept this refined was later deleted outright, but the tiering principle and replay argument carried forward into everything below.
- **`5c34279` — Implement the frozen-inputs / recorded-tooling split in the controller.** Deleted the controller-drift system (`PARETO_RAIL_ACCEPT_CONTROLLER_DRIFT`, frozen controller-admin hashes) and runner/executor pinning. Surviving pieces: the manifest records the executing controller commit; the scope check runs from the controller checkout; moving a run's `gates/` directory aside and resuming re-runs gates ("regate") with the disposition recomputed. Also the first test coverage for the runner's verification logic.
- **`fd0e645` — Isolate entrant checkouts.** An entrant now works in a standalone repository containing exactly the baseline commit (created by a depth-1 fetch of a temporary ref) instead of a worktree sharing the whole object database. Sealed commits and recovery snapshots are fetched back into the main repository under the existing names. Contamination control became a property of the setup; the "non-adversarial access" policy prose becomes deletable. Historical worktree-based runs still resume and prune (layout detection).
- **`350e31b` — Delete the protocol-record machinery; runs come from one private plan.** The freeze + cryptographic-schedule + hash-pinned-definition triad is gone: deleted `schedule.mjs` (441 lines), `run-schedule.schema.json`, `validateEligibleControls`, freeze-tag reading, per-artifact sha256 pre-verification, and the rehearsal/eligible mode split. A run is now selected by `--plan benchmark/private/v2-plan.json --run <runId>` from a hand-editable plan (materials commit, entrant baseline, one row per run with theme/recipe/stage; `kind: rehearsal` keeps smoke runs out of the results pool). Inputs are read from the materials commit and their hashes are recorded in the manifest (capture) instead of pre-checked. Added the render byte-identity pre-check across sibling runs of a theme.

## 5. Simplification phases (all landed)

- **`6dd9907` — Phase 2: delete the v1 dual code paths.** `migrate.mjs`, `inventory.mjs`, `entrant-baseline.mjs`, `check-benchmark-baseline.mjs`, and all `protocolForVersion` dispatch removed. `protocol.mjs` exports the two surviving footprints by name (`benchmarkLevelFootprint`, `builtInLevelFootprint`); `check-benchmark-scope.mjs` selects by invocation form (positional = built-in, flags = benchmark) with no version strings. `admin.mjs` always uses the benchmark footprint and requires the entrant-authored `level.json`. `promote.mjs` is a v2 publisher reading the flat run-definition shape and gating on `kind: benchmark`, keeping its lock, checkpoints, and working-tree guards.
- **`4271976` — Phase 3: slim the consumers.** `results.mjs` and `manage-run.mjs` are blind by default — run ids and dispositions only — with one `--unblind` flag; historical v1 records stay readable through flat-with-fallback definition reads. `export-rank-catalog.mjs` keeps the v1 schedule slice and reads `benchmark/private/v2-plan.json` when it exists, skipping rehearsal rows (`cf0f858` additionally withholds a plan slice with no public entrants so an unlaunched plan cannot claim the active pool). The freeze and ranking schemas are deleted; `run-manifest.schema.json` is slimmed to what the runner writes.
- **`784f056` — Phase 4: rewrite the documentation.** The runbook and failure taxonomy are deleted; `benchmark/controller/README.md` is the single operations page (run command, plan-file shape, resume/recovery, two-sentence failure policy, regate, run management, promotion, catalog publishing). `benchmark/README.md` is rewritten around the surviving system with the Cost section intact; `releases/README.md` shrinks to the tag pointer and plan-pinning rule; `manifests/`, `rankings/`, and `analysis/` carry one-line historical notes. `c91f7ed` retargets recipe references from the deleted runbook.
- **Phase 5: proven end to end.** Smoke rehearsal `smoke-v2-8q3k` (`pi-luna-low-smoke`, `kind: rehearsal`) ran the real pipeline: plan row → isolated checkout → stage → seal → four gates passed → payload → manifest with measured cost ($0.09, ~4 minutes wall). It surfaced one real bug — the renderer rejected the real template's repeated `{{LEVEL_TITLE}}` (`1d9e1e9`) — and the failed attempt is captured in the run record per the failure policy. Regate, the blind and unblinded listings, run status, and promotion's rehearsal refusal were all exercised against the finished run.

Everything past phase 5 — cutting the v2 entrant baseline, choosing themes and configurations, authoring the plan — is launch preparation, decided by the owner now that the simplified system is proven.

## 6. What deliberately survives

One deterministic run command with checkpointed resume; isolated entrant checkouts; sealed evaluated commits; four replayable gates; payload derivation; measured cost with reconciliation; budget-mode notices and continuations; recovery snapshots; opaque level ids with one private mapping; the site's blind pairwise voting. That is the whole benchmark — everything else was apparatus.
