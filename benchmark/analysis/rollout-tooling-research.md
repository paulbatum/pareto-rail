# Rollout Analysis & Visualization Tooling — Research

Survey of off-the-shelf approaches for analyzing and visualizing AI agent rollouts (transcripts/traces of long agentic coding sessions), commissioned to inform the design of a bespoke data format for the level-generation benchmark. Our rollouts are Claude Code sessions with subagents that build a game level unattended over ~1 hour, producing JSONL transcripts (a main session plus ~38 subagent transcripts), tool calls, file edits, screenshots (from the snapshot tooling), and token/cost usage. The target UI is an annotated replay with semantic sections and summaries, a delegation graph, file-edit history, and screenshots that change as edits land.

Every factual claim below was verified by fetching the cited page. Claims that rest only on secondary/search-snippet sources are flagged as unverified.

## TL;DR recommendation

Model our format on **UK AISI's Inspect AI `.eval` log**, not on OpenTelemetry GenAI or a from-scratch schema, and keep a documented mapping to OpenTelemetry/OpenInference vocabulary for later interoperability. Inspect is the only surveyed option whose data model already encodes almost everything we need — a typed event transcript, nested spans that group subagent and in-tool activity, tool-call/result events, sandbox events, per-model token usage, and screenshots as deduplicated `attachment://` references — and it is a de-facto standard that Anthropic's Petri, HuggingFace, and Transluce's Docent all consume ([eval-logs](https://inspect.aisi.org.uk/eval-logs.html), [log reference](https://inspect.aisi.org.uk/reference/inspect_ai.log.html), [agent-custom](https://inspect.aisi.org.uk/agent-custom.html)). Crucially, its open-source viewer bundles to a **self-contained static site** you can point at your own logs (`inspect view bundle`), so emitting Inspect-shaped JSON gets us a working viewer for free to validate the data before we build our own ([log-viewer](https://inspect.aisi.org.uk/log-viewer.html)). No observability platform (LangSmith, Langfuse, Braintrust, Weave, Phoenix, AgentOps, Helicone, PromptLayer) ships a drop-in embeddable trace-viewer component — every one is either closed or a whole self-hostable app — so viewer reuse effectively narrows to Inspect's static bundle. Two needs sit outside *every* existing format and are our genuine value-add: a first-class **file-edit history** (all tools model edits only as tool-call payloads) and **screenshots that swap as edits land** (no format ties a media timeline to edit versions). The precedent to follow is Petri: it reused Inspect's format but wrote its own viewer for bespoke visual needs, which is exactly our situation ([Petri](https://alignment.anthropic.com/2025/petri/)). Because our agent is Claude Code, note that the raw session JSONL *already* carries tool calls, `isSidechain` subagents, token/cost, and a versioned `file-history-snapshot` edit trail — much of the payload exists before we invent anything, so the format work is mostly transformation and enrichment, not capture ([Claude Code JSONL structure](https://gist.github.com/samkeen/dc6a9771a78d1ecee7eb9ec1307f1b52)). Net: **Inspect-log-shaped, bespoke-extended, with an OpenInference mapping** — reuse the schema and the throwaway viewer, build the annotated replay UI ourselves.

## Data-model lessons — concepts that recur everywhere

These primitives show up across nearly every tool surveyed. Our format should adopt them natively so it stays convertible to standards later, even though our value-add lives on top of them.

1. **A tree/DAG of nested steps is universal.** Whether called spans (OTel, Phoenix, AgentOps, Braintrust, PromptLayer), runs (LangSmith), observations (Langfuse), calls (Weave), or events-within-spans (Inspect), everyone represents a rollout as parent-child-nested units of work. **Adopt an explicit nesting primitive with stable parent IDs**, not a flat list. Inspect's *spans-wrapping-an-event-stream* is the most expressive variant: a flat ordered transcript for replay, with span begin/end markers delimiting subagent and in-tool activity for the delegation graph.

2. **Typed nodes with a small vocabulary.** Everyone tags each node with a kind — `llm`/`chat`, `tool`/`execute_tool`, `agent`/`invoke_agent`, `retriever`, `evaluator`. **Mirror this vocabulary** (align to OpenInference span kinds and OTel `gen_ai.operation.name` values) so exports map cleanly. Add our own kinds (e.g. a file-edit event, a screenshot event) as extensions rather than overloading tool nodes.

3. **Sessions/threads group multi-turn and multi-agent work.** Langfuse Sessions, Weave Threads, AgentOps SESSION roots, Inspect subtask spans, Claude Code `isSidechain`+`agentId`. **Model each subagent as a first-class session/span linked to the parent tool call that spawned it.** Design the linkage up front — parsers of Claude Code and Codex JSONL repeatedly hit an *orphaned-subagent ordering* problem where the subagent file appears before the parent's `tool_use` is written, requiring `agentId → tool_id` rematching.

4. **Attachments are references, not inline blobs.** The mature formats (Inspect `attachment://` dedup pool, Langfuse `mediaId`, Braintrust S3 links, LangSmith `Attachment`, OTel's recommended external-blob-with-span-reference pattern) all **store media out-of-band and keep a reference in the transcript**. Do the same for our screenshots — reference them, dedup them, and (our extension) key each reference to a file-edit version so the replay can show the render evolving.

5. **Scores/annotations are a distinct, attachable layer.** Langfuse Scores, Braintrust score/review spans, Weave evals, Inspect `Score`/`ScoreEvent`, Docent rubrics. **Keep semantic sections, summaries, and annotations as a separate overlay** attached by node/span id and time range, not baked into the step records — this is exactly how Docent and Petri layer human/LLM analysis over an immutable transcript.

6. **Immutable transcript + derived views.** The strongest precedent (Petri, Docent) treats the raw transcript as immutable and builds analysis/annotation/branch views on top. **Our annotated replay should be a derived view over an immutable rollout log**, so re-annotation never mutates source data.

7. **Token/cost is captured as usage; dollar cost is derived.** Every tool records per-call token usage (`gen_ai.usage.*`, `model_usage`, `message.usage`); none stores dollars natively. **Store token usage per node and derive cost** at display time.

Two things recur in the *gaps*: no surveyed format has a first-class **file-edit history** (edits are always just tool-call payloads), and none ties a **screenshot timeline to edit versions**. These are the parts we genuinely invent.

## Concrete recommendation: Inspect-log-shaped, bespoke-extended, with an OpenInference mapping

**Do not make the format OTel-GenAI-shaped.** The OpenTelemetry GenAI semantic conventions are still officially "Development"/experimental as of mid-2026 — moved into a dedicated `semantic-conventions-genai` repo with unstable-status badges — and they are optimized for *observability of individual LLM/tool calls*, not for narrative rollouts ([OTel spans doc](https://raw.githubusercontent.com/open-telemetry/semantic-conventions-genai/main/docs/gen-ai/gen-ai-spans.md), [OTel GenAI blog May 2026](https://opentelemetry.io/blog/2026/genai-observability/)). They model subagents only as span parentage (no session/handoff concept), have no file-operation semantics, and defer multimodal/attachment handling ([OTel agent-spans doc](https://raw.githubusercontent.com/open-telemetry/semantic-conventions-genai/main/docs/gen-ai/gen-ai-agent-spans.md)). Every one of our four higher-order needs falls outside it. It is the right thing to *map to*, not to *build on*.

**Do make it Inspect-log-shaped.** Inspect's `EvalSample` model already provides, natively: an ordered typed **event transcript** (`ModelEvent`, `ToolEvent`, `SandboxEvent`, `ScoreEvent`, …); **spans** auto-opened for subtasks, tool calls, and agent execution that group a subagent's entire activity as a nested, delimited region (with tool calls carrying their own in-tool sub-transcript); per-model **token usage**; a **`store`** for custom state; and an **`attachments`** pool that deduplicates images referenced by `attachment://` URIs, with `ContentImage` as a first-class message content type ([log reference](https://inspect.aisi.org.uk/reference/inspect_ai.log.html), [agent-custom](https://inspect.aisi.org.uk/agent-custom.html), [eval-logs](https://inspect.aisi.org.uk/eval-logs.html)). The format has a **published JSON schema** (`inspect log schema`), interconverts losslessly between `.eval` and `.json` (`inspect log convert`), and is the ecosystem's gravity well — Anthropic's Petri runs on it and Transluce's Docent ingests it directly, so an Inspect-shaped log is portable into real analysis tooling for free.

**The viewer argument seals it.** Across all eight observability platforms, *no one ships an embeddable trace-viewer component* — they are closed (LangSmith, Braintrust, PromptLayer) or whole self-hostable apps (Langfuse `/web`, AgentOps, Helicone, Phoenix iframe). Inspect is the exception: `inspect view bundle` produces a **self-contained static site** (`index.html` + `assets/`) you can point at your own log directory and deploy anywhere ([log-viewer](https://inspect.aisi.org.uk/log-viewer.html)). Emitting Inspect-shaped JSON therefore gives us a **zero-cost validation viewer** immediately, long before our own UI exists.

**Where we extend the schema (our value-add), layered so exports stay clean:**
- **File-edit history as a first-class event type** alongside `ToolEvent`, carrying path, version, and a content/diff reference. Since our agent is Claude Code, seed this directly from the native `file-history-snapshot` records (`trackedFileBackups`, `version`, `backupTime`) that already give us a versioned edit trail ([Claude Code JSONL structure](https://gist.github.com/samkeen/dc6a9771a78d1ecee7eb9ec1307f1b52)).
- **A screenshot event keyed to file-edit versions**, stored as an `attachment://`-style reference in the same dedup pool — this is the one thing no surveyed format does (screenshots that swap as edits land), and the reason the UI is bespoke.
- **A semantic-annotation overlay** (sections, summaries, delegation-graph metadata) attached by span/event id and time range, kept separate from the immutable transcript — the Docent/Petri pattern.

**Keep an explicit OpenInference mapping.** Document, per node type, the corresponding OpenInference span kind and `gen_ai.*`/`llm.*` attribute names (`execute_tool`, `invoke_agent`, `gen_ai.input/output.messages`, `gen_ai.usage.*`). OpenInference is the most usable open GenAI vocabulary — vendor-neutral, valid OTLP, and accepted as ingest by LangSmith and Weave ([OpenInference spec](https://github.com/Arize-ai/openinference/blob/main/spec/semantic_conventions.md), [OpenInference intro](https://arize-ai-openinference.mintlify.app/introduction)). This mapping costs us little now and keeps the door open to exporting our rollouts to standard backends or feeding Docent for cross-run analysis later.

**Precedent that this is the right shape of bet:** Petri adopted Inspect's *framework and log format* but wrote its *own* transcript viewer because it needed views Inspect's viewer couldn't do — multi-perspective panes, side-by-side branch visualization, inline citation highlighting ([Petri](https://alignment.anthropic.com/2025/petri/)). Our situation is identical: we need a delegation graph, file-diff-over-time, cost overlays, and screenshots synced to edits. So the plan is Petri's plan — **reuse the format, reuse the throwaway static viewer to validate, build the annotated replay UI ourselves** — with the SWE-agent `mini-traj-web-browser` static-site + auto-discovery + deep-linking pattern and `claude-code-trace`'s subagent-expand architecture as UI references ([mini-traj-web-browser](https://github.com/SWE-agent/mini-traj-web-browser), [claude-code-trace](https://github.com/delexw/claude-code-trace)).
